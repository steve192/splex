# Error handling

Splex exposes API failures using one JSON contract:

```json
{
  "error": {
    "code": "group_not_settled",
    "message": "Settle up before deleting this group.",
    "params": {}
  },
  "fields": {}
}
```

`message`, `params`, and `fields` are optional. `code` is always present for
errors created by the current API path.

## Two error-code families

- `backend/src/splex/shared/errors.py:ErrorCode` contains Splex business and
  user-action errors. Services raise `DomainError` or `DomainPermissionError`
  with one of these codes. This module has no DRF dependency, so it is safe to
  import from services and selectors.
- `backend/src/splex/shared/api.py:ApiErrorCode` contains framework failures
  generated by Django REST Framework, such as throttling, serializer
  validation, authentication, permission, and not-found errors. The
  `ERROR_CODES_BY_EXCEPTION` registry maps those exception classes at the HTTP
  boundary.

Both families serialize to the same public `error.code` field. Keeping them
separate prevents domain code from depending on DRF and prevents framework
details from leaking into services.

## Shared normalization model

`NormalizedApiError` is the single transport-neutral representation of the
public envelope. `ApiResponseError` is the base class for Splex exceptions that
can expose such a failure. `DomainError` and `DomainPermissionError` inherit
from it and provide `to_normalized_error()`.

The global handler, rather than an exception, constructs the DRF `Response`.
Native DRF exceptions are not wrapped: after DRF creates its response, the
framework adapter converts only its known public code, throttle parameters, and
validation fields to the same `NormalizedApiError`. It never forwards DRF
exception detail text, which may originate in a dependency.

## Unexpected internal failures and logging

Do not catch an unexpected programming, database, or dependency failure just
to make it an API error. The global handler returns `None` for it, so DRF
re-raises the exception and Django produces the 500 response.

This is intentional: Django logs the uncaught request through the
`django.request` logger at `ERROR` level with full exception information and a
stack trace. Production runs with `DEBUG = False`, so that diagnostic is kept
in server logs and is not returned to the client. A background task or command
does not use the API handler and must log its own unexpected failures.

## Adding a new backend error

### Splex business rule or expected user action

Use a domain error when a service can reject an otherwise valid request for a
known product reason: a group is not settled, a receipt is too large, a login
token expired, or an import credential was rejected.

1. Add an `ErrorCode` member in `backend/src/splex/shared/errors.py`.
2. Raise `DomainError(code, safe_message, status=..., params=...)` in the
   service, or `DomainPermissionError` for a 403. Keep `params` to primitive
   values used for translation interpolation.
3. Do not catch that error in an API view and return `Response({"detail": ...})`.
   Let `splex.shared.api.exception_handler` serialize it.
4. Add a regression test that asserts the API code, not only the HTTP status.

The message is sent to the client. It is a fallback for unknown codes and
non-Splex API consumers, so it must be a reviewed literal. Do not pass
``str(exc)`` or include secrets, tokens, raw provider responses, SQL details,
or stack traces. Catch an expected provider exception only to raise a domain
error with a fixed public message; leave unexpected internal exceptions
unhandled so Django returns a server error.

### Django REST Framework behavior

Do not create a domain code for standard DRF behavior such as throttling,
serializer field validation, authentication, or a normal 404. Add or adjust
the mapping in `ApiErrorCode` / `ERROR_CODES_BY_EXCEPTION` in
`backend/src/splex/shared/api.py` instead. Add a handler test for the mapped
exception.

Framework responses expose no `error.message`: the frontend translates the
known code. Validation errors retain their `fields` payload and throttling
retains `wait_seconds`. An unrecognised DRF `APIException` uses the generic
`api_error` code without forwarding its detail text.

### Manual responses

Do not return manual JSON error responses. A JSON endpoint with a controlled
failure must raise a domain or framework exception so it reaches the global
exception handler and has a stable code.

Manual non-JSON responses remain appropriate for endpoints that intentionally
return files, HTML, redirects, or another protocol-specific body. Their failure
paths must still raise a domain or framework exception.

## Frontend display and translations

`frontend/src/shared/api/client.ts` parses the API envelope into `ApiError`
with `code`, `message`, `params`, `status`, and the raw response data.

All user-facing error display must go through one of:

- `apiErrorMessage(error, t)` for reads or neutral actions;
- `apiWriteErrorMessage(error, t)` for writes, which gives offline failures a
  write-specific message;
- `apiErrorDescriptorMessage(descriptor, t, { write: true })` for persisted or
  deferred errors, such as sync and push registration.

Do not inspect `error.data.detail`, concatenate `error.message` in a screen,
or stringify a response object at a call site.

To make a code display translated copy, add one entry to the
`API_ERROR_TRANSLATION_KEYS` registry in
`frontend/src/shared/lib/apiErrors.ts`, pointing to an i18n key. Reuse an
existing semantic key where it accurately describes the action; otherwise add
a new source key to `frontend/src/shared/i18n/locales/en.json` and German copy
to `de.json`. The normal repository translation workflow fills the other
locales later; when an all-languages translation pass is explicitly requested,
update every locale and run `frontend/src/shared/i18n/locale.test.ts`.

Errors that occur before an HTTP response exists (offline, timeout, native file
upload failures) have no backend code. Create or preserve a local `ApiError`
and route it through the same frontend helpers.

## Request paths

```text
DomainError / DomainPermissionError
  -> exception_handler
  -> { error: { code, message, params } }
  -> ApiClient
  -> apiErrors registry
  -> translated UI text

DRF exception (Throttled, ValidationError, NotFound, ...)
  -> ERROR_CODES_BY_EXCEPTION
  -> { error: { code, params }, fields? }
  -> same frontend path

Unexpected internal exception
  -> exception_handler returns None
  -> DRF re-raises it
  -> Django logs the full traceback and returns a 500

Offline / timeout / native failure
  -> local ApiError in ApiClient
  -> frontend helper fallback; no backend code exists
```
